# Introduction to MD5

MD5, short for **Message-Digest Algorithm 5**, is a legacy cryptographic hash function used to convert data into a fixed 32-character hexadecimal string (a digital fingerprint).

### Key Characteristics:
* **Fast Performance:** MD5 is extremely fast compared to modern hashing algorithms like SHA-256.
* **Security Status:** It is considered **deprecated and insecure** for sensitive cryptographic purposes. This is due to discovered vulnerabilities that allow **Collisions** (where two completely different files or inputs produce the exact same hash value).
* **Current Use Cases:** While it is no longer secure for passwords or high-security applications, it is still commonly used in low-risk scenarios, such as verifying data integrity (**Checksums**) to ensure a file wasn't corrupted during a basic network transfer

❓ Required Lab Questions & Expected Answers

Question 1: If the length of your prefix file is not a multiple of 64, what is going to happen?

* Answer: MD5 processes data in blocks of 64 bytes. If the prefix length is not a multiple of 64, `md5collgen` will automatically add padding (zero-bytes) to fill up the current 64-byte block before appending the 128 bytes of collision data ($P$ and $Q$). As a result, the final file size will expand to the next available block boundary.

Question 2: Create a prefix file with exactly 64 bytes, and run the collision tool again, and see what happens.

* Answer: When the prefix file is exactly 64 bytes, it perfectly fills one complete MD5 block. The tool does not need to pad the initial block. It will directly append the 128 bytes of collision data. The final size of `out1.bin` and `out2.bin` will be exactly 192 bytes ($64 \text{ bytes prefix} + 128 \text{ bytes collision data} = 192 \text{ bytes}$).

Question 3: Are the data (128 bytes) generated by `md5collgen` completely different for the two output files? Please identify all the bytes that are different

Answer: No, the 128 bytes of generated data are not completely different. When inspecting `out1.bin` and `out2.bin` side-by-side in the `bless` hex editor, you will observe that they are almost identical. Only a few specific bytes (typically a few bits within specific bytes of the structural blocks) are altered to achieve the same hash value while keeping the rest of the generated block identical.


# Task 2: Understanding MD5's Property & Vulnerability

## Overview
In this task, we demonstrate a fundamental property of the MD5 hashing algorithm (and similar Merkle–Damgård hash functions):

> **Property:** Given two inputs $M$ and $N$ that collide under MD5 (i.e., $\text{MD5}(M) = \text{MD5}(N)$), appending the exact same suffix $T$ to both inputs results in two new inputs that also collide:
> $$\text{MD5}(M \parallel T) = \text{MD5}(N \parallel T)$$

---

## Important Clarification regarding $M$ and $N$

**Can $M$ and $N$ be arbitrary random text files created manually?**
- **NO.** If you type two random text files manually (e.g., `file1.txt` containing "hello" and `file2.txt` containing "world"), their initial MD5 hashes will **NOT** be equal ($	ext{MD5}(M) \neq \text{MD5}(N)$).
- Consequently, appending suffix $T$ to both will **NOT** yield matching hashes.
- **Requirement:** $M$ and $N$ **MUST** be two files that already have identical MD5 hash values (a collision pair generated using a tool like `fastcoll` or provided in Task 1).

---

## Detailed Step-by-Step Procedure & Commands

### Step 1: Obtain the Prefix Collision Pair ($M$ and $N$)
Generate two different binary/text files $M$ and $N$ that share the exact same MD5 hash using `fastcoll`:

```bash
# Generate two colliding files prefix_out1.bin (M) and prefix_out2.bin (N)
fastcoll -o prefix_out1.bin prefix_out2.bin
```

Verify that $M$ and $N$ have the same MD5 hash:
```bash
md5sum prefix_out1.bin prefix_out2.bin
```
*(Both outputs should display identical 128-bit hash strings).*

---

### Step 2: Create the Suffix File ($T$)
Create a third file $T$ containing any arbitrary text or binary data:

```bash
# Create suffix file T with arbitrary content
echo "This is the common suffix data T appended to both files." > suffix_T.txt
```

Verify the contents and hash of $T$:
```bash
cat suffix_T.txt
md5sum suffix_T.txt
```

---

### Step 3: Concatenate $T$ to $M$ and $N$
Use the Linux `cat` command to append suffix $T$ to file $M$ ($M \parallel T$) and to file $N$ ($N \parallel T$):

```bash
# Append T to M (prefix_out1.bin) -> output M_new.bin
cat prefix_out1.bin suffix_T.txt > M_new.bin

# Append T to N (prefix_out2.bin) -> output N_new.bin
cat prefix_out2.bin suffix_T.txt > N_new.bin
```

---

### Step 4: Verify the Hashes of the New Combined Files
Compute the MD5 checksums for `M_new.bin` and `N_new.bin`:

```bash
md5sum M_new.bin N_new.bin
```

### Expected Result
Even though `M_new.bin` and `N_new.bin` differ in their initial prefix blocks ($M \neq N$), their resulting MD5 hash values will be **100% identical**.

---

## Why Does This Happen? (Theoretical Background)

MD5 operates on 64-byte blocks iteratively using a compression function $f$:
1. The first block $M_1$ is processed with a fixed initial hash value $\text{IHV}_0$: 
   $$\text{IHV}_1 = f(\text{IHV}_0, M_1)$$
2. Once the initial colliding prefix files $M$ and $N$ are processed, they yield the exact same intermediate hash value:
   $$\text{IHV}_M = \text{IHV}_N$$
3. When suffix $T$ is appended, the compression function processes $T$ starting from this identical intermediate state ($\text{IHV}_M = \text{IHV}_N$).
4. Therefore, the resulting final hash $\text{IHV}_{final}$ must be identical for both $M \parallel T$ and $N \parallel T$.
