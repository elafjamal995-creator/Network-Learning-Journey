# Introduction to MD5

MD5, short for **Message-Digest Algorithm 5**, is a legacy cryptographic hash function used to convert data into a fixed 32-character hexadecimal string (a digital fingerprint).

### Key Characteristics:
* **Fast Performance:** MD5 is extremely fast compared to modern hashing algorithms like SHA-256.
* **Security Status:** It is considered **deprecated and insecure** for sensitive cryptographic purposes. This is due to discovered vulnerabilities that allow **Collisions** (where two completely different files or inputs produce the exact same hash value).
* **Current Use Cases:** While it is no longer secure for passwords or high-security applications, it is still commonly used in low-risk scenarios, such as verifying data integrity (**Checksums**) to ensure a file wasn't corrupted during a basic network transfer

## 🛠️ Commands & Explanations

Run these commands in your SEED Ubuntu terminal to execute the lab:

1. **Create a 64-byte prefix file:**
   ```bash
   echo -n "1234567890123456789012345678901234567890123456789012345678901234" > prefix.txt
   ```
   * Explanation: This creates a text file named prefix.txt containing exactly 64 characters (64 bytes)
   ## 1- Generate the MD5 Collision files:
```text
md5collgen -p prefix.txt -o out1.bin out2.bin
```
* Explanation: Runs the collision tool using prefix.txt as the base. It outputs two different binary files: out1.bin and out2.bin.
## 2- Check the content difference:
```text
diff out1.bin out2.bin
```
* Explanation: Compares the two files. It will show that they have different contents.
## 3-Verify the MD5 Hashes:
```text
md5sum out1.bin
md5sum out2.bin
```
* Explanation: Displays the MD5 hash for both files. You will see that the hashes are 100% identical, proving the collision.

## 4-Open the Hex Editor to inspect bytes:
```text
bless out1.bin &
bless out2.bin &
```
* Explanation: Opens both binary files side-by-side in the bless Hex Editor to visually find which bytes are different.

❓ Required Lab Questions & Expected Answers

Question 1: If the length of your prefix file is not a multiple of 64, what is going to happen?

* Answer: MD5 processes data in blocks of 64 bytes. If the prefix length is not a multiple of 64, `md5collgen` will automatically add padding (zero-bytes) to fill up the current 64-byte block before appending the 128 bytes of collision data ($P$ and $Q$). As a result, the final file size will expand to the next available block boundary.

Question 2: Create a prefix file with exactly 64 bytes, and run the collision tool again, and see what happens.

* Answer: When the prefix file is exactly 64 bytes, it perfectly fills one complete MD5 block. The tool does not need to pad the initial block. It will directly append the 128 bytes of collision data. The final size of `out1.bin` and `out2.bin` will be exactly 192 bytes ($64 \text{ bytes prefix} + 128 \text{ bytes collision data} = 192 \text{ bytes}$).

Question 3: Are the data (128 bytes) generated by `md5collgen` completely different for the two output files? Please identify all the bytes that are different

Answer: No, the 128 bytes of generated data are not completely different. When inspecting `out1.bin` and `out2.bin` side-by-side in the `bless` hex editor, you will observe that they are almost identical. Only a few specific bytes (typically a few bits within specific bytes of the structural blocks) are altered to achieve the same hash value while keeping the rest of the generated block identical.
